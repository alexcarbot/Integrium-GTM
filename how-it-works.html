PLATFORM ARCHITECTURE

How Integrium CORE processes wound care data

Four architectural layers transform clinical terminology into validated billing claims — every step deterministic, every decision traceable.

The Architecture

Four layers from terminology to validated claim

Each layer is independently testable and produces traceable output. No black boxes, no probabilistic guesses.

1

Terminology Service

Loads 715K+ concepts from SNOMED CT, ICD-10-CM, CPT, and HCPCS authoritative distributions. Complex map evaluation handles context-dependent translations for wound care diagnoses.

2

Rules Engine

12 deterministic billing endpoints backed by PostgreSQL. Each endpoint evaluates wound care billing rules — NCCI edits, LCD requirements, CPT selection — and returns a RuleTrace audit record.

3

Clinical Workflow

FHIR R4 profiles for wound assessment panels, dual-coded conditions (SNOMED + ICD-10), and structured clinical documentation that feeds directly into the billing pipeline.

4

Billing Validation

CPT calculation, NCCI conflict detection, LCD gap analysis, and sign-and-lock checks run on every encounter. CMS quarterly refresh pipeline keeps all rules current with effective dates.

Design Principles

Why deterministic rules, not AI

Billing compliance demands explainability and auditability. Every decision traces to a specific rule, not a probability score.

FHIR R4 Native

Clinical data is stored and exchanged using FHIR R4 resources from the start — Condition, Observation, Procedure, and Claim profiles purpose-built for wound care workflows.

Full RuleTrace

Every billing endpoint returns a complete trace of which rules evaluated, what data was considered, and why the decision was made. No black boxes — every output is auditable.

CMS Quarterly Refresh

Updated NCCI edit tables, LCD changes, and code set revisions are ingested from authoritative CMS distributions. Version-controlled rules with effective dates keep validation current.

Data Flow

How data moves through the platform

From clinical documentation through terminology translation to validated billing claim — each stage is traceable and auditable.

Layer 1

Clinical Data Capture

  • Wound assessment panels capture size, depth, tissue type, and location
  • FHIR R4 Observation and Condition resources store structured clinical data
  • Dual-coded conditions maintain both SNOMED CT and ICD-10-CM representations
  • Clinical documentation feeds directly into the terminology service
Layer 2

Terminology Translation

  • 715K+ concepts from authoritative SNOMED CT, ICD-10-CM, CPT, and HCPCS distributions
  • Complex map evaluation handles laterality, specificity, and context-dependent mappings
  • Cross-coding between clinical terms and billing codes uses deterministic rules
  • Translation output feeds into CPT calculation and billing validation
Layer 3

Rules Engine Processing

  • 12 deterministic billing endpoints evaluate NCCI edits, LCD rules, and CPT selection
  • PostgreSQL-backed rule storage with version control and effective dates
  • 106 dual-reviewed billing scenarios validated against CMS guidelines
  • Every rule evaluation produces a complete RuleTrace audit record
Layer 4

Validated Claim Output

  • Sign-and-lock gate validates ICD-10 specificity, laterality, and dx pointer order
  • E&M level, smoking status, and POA indicators checked before release
  • CMS quarterly refresh pipeline ingests updated rules from authoritative distributions
  • 684+ automated tests verify engine behavior across the full validation pipeline
Technical Questions

Frequently asked questions

Integrium CORE is built natively on FHIR R4 (4.0.1). Clinical data is stored as FHIR resources from the start — Condition, Observation, Procedure, and Claim profiles are purpose-built for wound care workflows. The platform uses FHIR-native APIs for data exchange, not translation layers on top of proprietary formats.
The CMS quarterly refresh pipeline ingests updated NCCI edit tables, LCD revisions, and code set changes directly from authoritative CMS distributions. Rule updates are version-controlled in PostgreSQL with effective dates, so the engine always validates against the rules in effect at the time of service. No manual rule updates are required.
Every billing decision produces a RuleTrace record that includes: which rules fired, what clinical and billing data was evaluated, the pass/fail result of each validation step, the rule version and effective date, and a timestamp. RuleTrace records are immutable and exportable for payer audits, internal review, or compliance reporting.
Integrium CORE is designed as a multi-tenant cloud platform with PostgreSQL for rule storage and FHIR R4 for clinical data. The architecture supports tenant isolation for data security. Deployment details and hosting options can be discussed during a technical demo.
The terminology service loads 715K+ concepts from authoritative SNOMED CT, ICD-10-CM, CPT, and HCPCS distributions. When a clinician documents a wound care condition (e.g., diabetic foot ulcer), the complex map evaluation engine translates the clinical SNOMED term to the correct ICD-10-CM billing code, handling laterality, specificity, and context-dependent mappings deterministically.
